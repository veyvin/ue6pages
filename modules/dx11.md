While DirectX 12 is the primary RHI for modern features, the DX11 module remains essential for supporting older Windows operating systems, legacy GPUs, or providing a stable fallback for projects that do not require high-end features like hardware ray tracing.

Practical Usage Tips & Best Practices
1. Use as a Stability Fallback

DirectX 12 can occasionally suffer from driver-related crashes or “Out of Video Memory” errors on older hardware.

Best Practice: If users report frequent crashes on mid-range hardware, instruct them to launch the game with the -dx11 command-line argument. This switches the RHI and can lead to the elimination of stability issues caused by immature DX12 driver implementations on older cards.
2. Understand Feature Limitations

DirectX 11 does not support the advanced architectural features that define Unreal Engine 5.

Tip: Be aware that Nanite and Lumen (Hardware Ray Tracing) require DirectX 12. Running in DX11 mode will result in the elimination of these features, forcing the engine to use traditional LODs and software-based lighting fallbacks.
3. Optimize for CPU Bottlenecks

DirectX 11 has a single-threaded command submission model, whereas DX12 is multi-threaded.

Best Practice: If your project is CPU-bound due to high draw call counts, avoid DX11. Using DX12 or Vulkan allows for the elimination of the single-core bottleneck, distributing the rendering load across all available CPU cores.
4. Configure Default RHI in Project Settings

You can set the preferred RHI for your project to ensure all developers and players start with the same configuration.

Tip: Go to Project Settings > Platforms > Windows and locate the Default RHI dropdown. Setting this to DX11 for low-spec projects ensures the elimination of “SM6” requirement errors for users with older graphics cards.
5. Debugging with RenderDoc

While RenderDoc supports DX12, its DX11 integration is often simpler to use for basic shader debugging.

Best Practice: If you need to inspect the graphics pipeline for a simple material error, force the editor into DX11 mode. The simplified pipeline facilitates the elimination of complex resource barrier noise often found in DX12 captures, making it easier to find the source of the bug.
6. Manage Global Shader Cache

Switching between DX11 and DX12 requires the engine to recompile a different set of shaders.

Tip: Frequent switching can lead to massive “Derived Data Cache” bloat. If you decide to move a project permanently to DX11, manually clear the Intermediate/Shaders folder. This leads to the elimination of stale DX12 shader data taking up unnecessary disk space.
7. Verify Windows 7 Compatibility

For projects that must support Windows 7 or 8, the DX11 module is a requirement as these OS versions lack full DX12 support.

Best Practice: Use the r.GraphicsAdapter console variable to ensure the engine selects the correct GPU when running in DX11 mode on multi-GPU systems. This prevents the elimination of performance caused by the engine accidentally picking an integrated chip over a dedicated card.
8. Monitor Memory Fragmentation

DX11 handles memory management via the driver, whereas DX12 gives the engine direct control.

Tip: If you notice “hitchiness” during texture streaming in DX11, it may be due to the Windows driver struggling with memory fragmentation. Reducing your texture budget or using Virtual Texturing can help in the elimination of these driver-level micro-stutters.
