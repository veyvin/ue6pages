### AI Description & Usage Tips

e (RHI) implementation that allows Unreal Engine to communicate with Microsoft’s DirectX 11 graphics API.

Description and Purpose

This module serves as a compatibility layer for Windows-based systems that do not support or require the advanced features of DirectX 12. While DX12 is the modern standard for UE5 (enabling Nanite, Lumen, and Hardware Ray Tracing), the DX11 module remains essential for maintaining support for older GPUs and operating systems. It translates the engine’s platform-independent rendering commands into the Direct3D 11 API calls necessary for the graphics card to draw the frame. It is the primary fallback for projects where stability and broad hardware compatibility are prioritized over cutting-edge visual features.

Practical Usage Tips and Best Practices
Understand Feature Limitations
The DX11 module cannot utilize many of Unreal Engine 5’s flagship features. To eliminate visual inconsistencies, remember that Nanite and Hardware Ray Tracing are unavailable in DX11 mode. Nanite meshes will fall back to their proxy meshes, and Lumen will use significantly less accurate software ray tracing or screen-space techniques.
Force DX11 via Command Line
If you need to test your game’s performance on older hardware or troubleshoot driver crashes, you can force the engine to use this module by adding -dx11 to your executable’s command-line arguments. This is a vital step to eliminate RHI-specific bugs during the QA process.
Monitor Shader Compilation Performance
DX11 handles shaders differently than modern APIs. If you experience long hitches during gameplay—especially during intense moments like an elimination explosion—check your log for shader compilation entries. Using the r.D3D11.PrecompileShaders console variable can help eliminate these stutters by forcing shaders to load earlier.
Configure Default RHI in Project Settings
You can set the default RHI for your project under Project Settings > Platforms > Windows. Setting this to DirectX 11 is recommended if your target audience uses older enterprise hardware. This helps you eliminate “Driver Not Supported” errors for users without DX12-compatible systems.
Optimize for Draw Call Overhead
DirectX 11 has a much higher CPU overhead for draw calls compared to DX12. To eliminate CPU bottlenecks in your project, use Instanced Static Meshes (ISMs) or Hierarchical Instanced Static Meshes (HISMs) more aggressively when the DX11 module is active.
Handle Device Loss Gracefully
The DX11 module can experience “Device Removed” errors if the GPU driver crashes or hangs. Ensure your code does not make assumptions about persistent GPU resources. Handling the GRHINeedsResubmit state will help you eliminate full application crashes when a minor driver reset occurs.
Toggle the RHI via Configuration Files
If the Editor will not open due to a DX12 crash, navigate to Config/DefaultEngine.ini and set DefaultGraphicsRHI=DefaultGraphicsRHI_DX11. This allows you to bypass the crash and eliminate the need to reinstall the engine or drivers to regain access to your project.
Check GPU Compatibility via CVars
Use the console command r.RHI.Name to verify that the DX11 module is active at runtime. This is the most reliable way to eliminate guesswork when debugging performance differences between different user machines or development environments.
