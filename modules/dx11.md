Description

In Unreal Engine 5, the DX11 module serves as the legacy fallback RHI for Windows systems. While DirectX 12 (DX12) is the primary engine default—required for modern features like Nanite, Lumen, and Virtual Shadow Maps—the DX11 module provides a stable environment for older hardware and operating systems (such as Windows 7 or older GPUs). It translates the engine’s platform-agnostic rendering commands into DX11-specific API calls. Because it lacks the low-level memory management and multi-threaded command list features of DX12, it is generally less performant on modern high-core-count CPUs.

Practical Usage Tips and Best Practices
1. Switching RHIs for Compatibility

If your project needs to support older hardware, you can change the default RHI in Project Settings > Platforms > Windows > Default RHI. Selecting DirectX 11 will invoke the DX11 module upon the next engine restart. This is a critical best practice for shipping titles intended for a broad “minimum spec” audience.

2. Understand Feature Limitations

Be aware that the DX11 module does not support Unreal Engine 5’s flagship features. Specifically:

Nanite: Will fall back to standard LODs or proxy meshes.
Lumen: Will fall back to Screen Space Reflections or simpler Global Illumination.
Hardware Ray Tracing: Is entirely disabled. Use the DX11 module only if your project is designed with a traditional “forward-plus” or “deferred” rendering pipeline in mind.
3. Debugging via Console Variables

To verify which RHI is currently active, use the console command r.RHI.Name. If the output is D3D11, the DX11 module is in control. For performance debugging, you can use stat d3d11rhi to see detailed statistics about draw calls and state changes specific to this module.

4. Manage Shader Model 5 (SM5)

The DX11 module is tied to Shader Model 5. When developing materials, ensure you do not use nodes that require SM6 (like certain wave intrinsics). The module will attempt to compile all shaders for SM5, and any incompatible logic will lead to “Default Material” (checkerboard) rendering in the viewport.

5. Optimization via Draw Call Elimination

Since DX11 has higher CPU overhead per draw call than DX12 or Vulkan, you must be more aggressive with Actor Merging and Instancing. The elimination of redundant draw calls is far more impactful in the DX11 module, as it cannot effectively parallelize command list generation across multiple CPU cores.

6. Use .ini Files for Emergency Fallbacks

If a user’s hardware cannot launch the engine in DX12, the DX11 module can be forced via the command line with -dx11. Alternatively, you can edit the DefaultEngine.ini file under [/Script/WindowsTargetPlatform.WindowsTargetSettings] by setting DefaultGraphicsRHI=DefaultGraphicsRHI_DX11. This is the standard way to recover a project that crashes on startup due to GPU driver incompatibilities.

7. Handle Driver Elimination and TDRs

DirectX 11 is susceptible to TDR (Timeout Detection and Recovery) crashes if a shader takes too long to execute. If the Windows OS “eliminates” the driver session to prevent a system hang, the DX11 module will crash the engine. Keep your post-process volumes and complex pixel shaders optimized to avoid hitting these OS-level hardware guards.

8. Monitor Memory Fragmentation

Unlike DX12, where the engine manages its own video memory (VRAM), the DX11 module relies more heavily on the Windows driver for memory allocation. This can lead to VRAM fragmentation over long play sessions. A best practice when using this module is to occasionally use the r.Streaming.PoolSize command to ensure the texture streamer isn’t over-allocating beyond what the driver can handle.
