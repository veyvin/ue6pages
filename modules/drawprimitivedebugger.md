### AI Description & Usage Tips

What it’s used for

Located in the Developer module category, this tool is designed to bridge the gap between high-level scene statistics and low-level RHI (Rendering Hardware Interface) profiling. It allows developers to identify exactly which assets are contributing to high draw call counts or high polygon density at runtime.

Primary uses include:

Draw Call Analysis: Counting the number of draw calls generated by a specific actor or component.
LOD Verification: Visually checking which Level of Detail (LOD) is currently active for a primitive and forcing specific LODs for testing.
Nanite Inspection: Toggling Nanite rendering on and off for specific meshes to compare performance and visual fidelity.
Asset Auditing: Identifying materials, textures, and bone counts associated with a specific rendered primitive.
Practical Usage Tips and Best Practices
1. Access via Console Command

To open the tool during a Development or Test build, use the console command: PrimitiveDebugger.Open This opens a dedicated UI window. Note that this tool is a runtime utility and is not available in “Shipping” builds to ensure the elimination of debug overhead in the final product.

2. Use Filters to Isolate Actors

In complex scenes, the primitive list can be overwhelming. Use the search bar at the top of the window to filter by Actor Name, Material Name, or Component Class. This allows you to focus on a specific area of the level, such as “Foliage” or “Enemies,” to see their collective impact on the frame.

3. Force LODs for Optimization Testing

The Details panel within the debugger provides a slider for non-Nanite meshes to force a specific LOD. Use this to verify that your distant LODs are actually being used and to visually confirm that the “pop” between LOD levels is acceptable at various distances.

4. Compare Nanite vs. Non-Nanite

For Nanite-enabled meshes, the debugger allows you to force-disable Nanite rendering for that specific instance. This is a best practice for troubleshooting material inconsistencies or verifying if Nanite is actually providing a performance benefit for a specific piece of geometry.

5. Visualize Skeletal Mesh Bones

In Development builds, the Primitive Debugger can toggle a real-time debug skeleton for skeletal meshes. This is invaluable for identifying rigging errors or animation glitches where bones might be stretching or incorrectly positioned, which can sometimes cause unexpected draw call spikes.

6. Pin Important Primitives

If you are iterating on a specific asset’s performance, use the Pin feature to keep that primitive at the top of the list. Even if you move the camera or refresh the scene, your pinned assets remain easily accessible for constant monitoring.

7. Check for Draw Call Bloat

If an actor shows a high number of draw calls in the debugger, check the Materials section. Each unique material ID on a mesh adds at least one draw call. Use this information to identify assets that need to be re-authored with combined textures or “atlases” to reduce the rendering burden.

8. Strategic Elimination of Hidden Costs

Use the debugger to find “Hidden” actors that are still contributing to the draw call count. If an object is set to Hidden in Game but its PrimitiveComponent is still being processed by the renderer, the Primitive Debugger will flag it, allowing you to properly disable the component and save performance.
