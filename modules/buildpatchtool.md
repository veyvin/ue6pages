It is the primary engine-side technology used by the Epic Games Launcher and custom launchers to perform delta updates. It works by breaking files into small, reusable data chunks, allowing the client to only download the specific data that has changed, which significantly reduces bandwidth costs and installation times.

Practical Usage Tips and Best Practices
1. Implement a Consistent Build Pipeline

The BuildPatchTool requires access to the “Cloud Directory” (where previous manifests and chunks are stored).

Best Practice: Always run the tool on a dedicated build machine that has persistent access to your historical build data. This ensures the tool can recognize existing chunks from previous versions, which helps eliminate redundant data generation and reduces the size of the patch.
2. Optimize Chunk Size for Your Audience

You can control the average size of the data chunks generated by the tool using the -ChunkSize parameter.

Tip: For games with very large assets (like 4K textures), larger chunk sizes can improve compression efficiency. However, for mobile or slow-connection targets, smaller chunks are preferred. Proper tuning helps eliminate wasted bandwidth during the update process.
3. Use the -Compact Command Regularly

Over many updates, the Cloud Directory can accumulate “orphaned” chunks that are no longer referenced by any active manifest.

Action: Periodically run the BuildPatchTool with the -mode=Compact argument. This will identify and eliminate unused data chunks from your storage, preventing your CDN costs from ballooning over time.
4. Automate Manifest Generation

The tool produces a .manifest file which acts as the “instruction manual” for the client’s patcher.

Best Practice: Use a naming convention for your manifests that includes the Build ID and Platform (e.g., MyGame_v1.2_Win64.manifest). Automating this via a Python or PowerShell script will eliminate human error when deploying new versions to your distribution platform.
5. Verify Builds Before Deployment

The tool includes a verification mode to ensure that the generated chunks and manifests are not corrupted.

Tip: Use -mode=Verify after generating a new build. This checks the integrity of the data in the Cloud Directory. Running this check helps eliminate “corrupt installation” errors for your end-users before the patch goes live.
6. Leverage Delta Patching for Frequent Updates

If you are running a Live Service game with weekly updates, binary deltas are essential.

Best Practice: Use the -mode=Diff command to compare two specific manifests. This allows you to see exactly how much data a player moving from Version A to Version B will have to download. Use this data to eliminate unnecessarily large files from your “hotfix” updates.
7. Secure Your Manifests

Manifest files contain a list of all files in your game and their hashes.

Tip: While the manifest is necessary for the patcher, do not leave it in a publicly browsable directory on your web server. Use secure signed URLs or an authentication layer to eliminate the risk of unauthorized users “scraping” your file structure via your manifest files.
8. Coordinate with Fab and Epic Games Store

If you are publishing on the Epic Games Store, the backend systems use this technology natively.

Action: Familiarize yourself with the BPT (Build Patch Tool) documentation in the Epic Developer Portal. Understanding how the tool chunks your specific project will help you structure your .pak files to eliminate large, monolithic updates that frustrate players.
