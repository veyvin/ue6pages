Description and Purpose

This module is primarily responsible for the “Visual Editing” of analytics providers. It provides Detail Customizations (custom UI layouts) for the Project Settings > Analytics category. Instead of editing raw configuration files, developers use the UI provided by this module to select active analytics providers, configure account keys, and toggle debugging features. It ensures that complex analytics data structures are represented clearly in the Details Panel, facilitating faster iteration for data-driven design teams.

Practical Usage Tips and Best Practices
Editor-Only Dependency
This module should never be included in a runtime or shipping build. When creating a custom editor tool that interacts with analytics settings, add it only to your Editor module’s Build.cs:
C#
	    if (Target.Type == TargetType.Editor)

	    {

	        PrivateDependencyModuleNames.Add("AnalyticsVisualEditing");

	    }

	    ```

	 

	*   **Verify Provider Configuration**  

	    Use the UI generated by this module (found in *Project Settings > Analytics*) to confirm that your `ProviderModuleName` is correctly mapped. This is the fastest way to check if the engine recognizes your custom analytics provider (like a CSV or cloud-based plugin).

	 

	*   **Custom Detail Customizations**  

	    If you are writing a custom analytics provider, you can look at the source code of `AnalyticsVisualEditing` (found in `Source/Editor/AnalyticsVisualEditing`) as a reference for how to write your own `IDetailCustomization` to expose provider-specific settings (like API keys or batch sizes) with custom validation.

	 

	*   **Debug Gameplay Events via Settings**  

	    When testing events such as a player **elimination**, use the settings provided by this module to toggle "Debug Logging" for your provider. This allows you to see the raw event data in the Output Log as it is being processed by the visual editing logic, ensuring that your attributes (e.g., `KillerID`, `WeaponType`) are being captured correctly.

	 

	*   **Standardize Analytics Settings**  

	    Leverage the module's support for `UAnalyticsSettings`. By using the standard analytics settings object, your custom tools can automatically benefit from the visual layout and property filtering that this module provides to the Unreal Editor.

	 

	*   **Avoid Runtime Logic in Visual Classes**  

	    Remember that classes within this module are meant for editing, not execution. Do not place gameplay logic—such as determining when an **elimination** occurs—inside customizations provided by this module. Keep your analytics *trigger* logic in runtime modules and use this module strictly for *configuring* the destination of that data.

	 

	*   **Isolate Slate UI logic**  

	    If you need to extend the analytics dashboard, use the Slate widgets provided by this module (like `SAnalyticsProviderDetails`) rather than building new ones from scratch. This maintains a consistent "Look and Feel" across the engine's data-tracking tools.

	 

	*   **Monitor Module Loading Phases**  

	    Since this module provides UI for settings, it typically loads during the `PostEngineInit` phase. If you are developing a plugin that needs to customize these settings, ensure your plugin's loading phase is set to `Default` or later to ensure `AnalyticsVisualEditing` is already initialized and ready to handle your customizations.
