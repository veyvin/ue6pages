### AI Description & Usage Tips

rt platform-native debug symbols into a format compatible with Google Breakpad. It is a critical component of the engine’s crash-reporting and symbolication pipeline.

Description

This module acts as a translation layer that reads debug information—such as Windows PDB files, macOS/iOS dSYM files, or Linux DWARF data—and encodes it into the .sym text format used by Google Breakpad. In the context of Unreal Engine, this tool is essential for teams that use custom crash-reporting backends or third-party services that rely on the Breakpad ecosystem to transform cryptic memory addresses into human-readable function names and file paths. It effectively bridges the gap between the raw binary crash dump and a developer-friendly call stack.

Practical Usage Tips and Best Practices
1. Integrate into CI/CD Pipelines

Automate the execution of this module within your Continuous Integration (CI) server. Every time a “Shipping” or “Test” build is generated, the BreakpadSymbolEncoder should run to convert the resulting symbols. This ensures that if a crash occurs in the wild, you already have the corresponding .sym files ready to identify the cause.

2. Strictly Manage Symbol Versioning

Debug symbols must match the specific binary that crashed. Always store your encoded .sym files alongside the specific build version they represent. If you attempt to symbolicate a crash dump with symbols from a different build, the resulting call stack will be inaccurate or completely unreadable.

3. Essential for Non-Windows Platforms

While Windows developers often use Microsoft’s Symbol Server, the BreakpadSymbolEncoder is particularly vital for Linux and Android development. These platforms frequently use Breakpad-based tools (like Sentry or custom Crash Report Servers) to handle signals and exceptions. Ensure this module is included in your Linux build targets to maintain visibility into server-side crashes.

4. Optimize Symbol Uploads

The .sym files generated by this module can be quite large, especially for complex projects. Use scripts to compress these files before uploading them to your symbol server. This reduces storage costs and eliminates excessive bandwidth usage during the build-and-deploy phase.

5. Verify Symbol Stripping

When packaging for platforms like Android, ensure that you are not “stripping” all debug information before the BreakpadSymbolEncoder has a chance to read it. The encoder requires the unstripped binary (containing the full DWARF/PDB information) to generate a valid .sym file. Once the encoding is complete, you can then safely strip the binary for the final distribution to eliminate unnecessary file size.

6. Debugging the “Unknown Function” Error

If your crash reports are showing “Unknown Function” or hex addresses (e.g., 0x0000000140001234), it usually means the encoder failed to find a specific symbol or the .sym file was not uploaded to the server. Check the encoder’s output log during the build process to ensure no “Symbol Not Found” warnings occurred.

7. Analyze Crashes on Elimination

In high-performance multiplayer games, crashes often occur during the destruction of actors—such as a player’s elimination or the cleanup of a large physics object. Use the symbolicated logs provided by Breakpad to track down “Use After Free” errors that frequently happen during the elimination of complex class hierarchies, where a pointer might be accessed after the actor has been removed from memory.

8. Use with Crash Report Client (CRC)

The BreakpadSymbolEncoder is often used in tandem with the engine’s Crash Report Client. While the CRC sends the report, the encoder ensures the backend server can actually make sense of it. If you are building a custom crash backend, refer to the source code in Engine/Source/Developer/BreakpadSymbolEncoder to understand how the engine structures the symbol metadata for proper alignment with the dump files.
