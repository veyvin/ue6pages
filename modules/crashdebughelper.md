What it is and What it’s used for

Located in the Developer section of the engine, this module acts as a bridge between raw binary crash data and human-readable reports. It interfaces with the Windows Debug Help (DbgHelp) library and symbol servers to resolve memory addresses into function names and source code lines.

Primary uses include:

Post-Mortem Analysis: Automatically extracting call stacks from crash dumps generated by the Crash Report Client.
Symbol Resolution: Matching a crash dump with the correct debug symbols (.pdb files) to identify the exact point of failure.
Automated Bug Reporting: Powering backend tools that aggregate similar crashes into groups to help developers identify the most frequent “elimination” events in their game.
Log Extraction: Pulling the final lines of the engine log from a crash report to provide context for the failure.
Practical Usage Tips and Best Practices
1. Integrate into CI/CD Pipelines

Use the ICrashDebugHelper interface to build a custom commandlet or standalone tool for your Build Farm. When a playtest build crashes, the helper can automatically parse the dump and post the call stack directly to a Slack or Discord channel, ensuring the elimination of delays in notifying the team.

2. Configure a Local Symbol Store

The module relies heavily on symbols. To use it effectively, ensure your build system uses SymStore.exe to push .pdb files to a central network share. Point the module to this path so it can automatically find the correct symbols for every build version ever released.

3. Use SetGameData for Context

Before a crash even happens, use FPlatformCrashContext::SetGameData in your game logic. The CrashDebugHelper can then extract these custom key-value pairs (like “CurrentLevel” or “PlayerHealth”) from the report, providing vital context that the call stack alone might miss.

4. Manage PDB Lifetime

Debug symbols are large. Use the helper’s ability to identify which builds are still “active” in the field. When the helper no longer sees crashes coming from an old version, you can safely archive or delete those specific symbols to save server space.

5. Handle Unattended Analysis

When using the helper in a server-side script, ensure you set it to “Unattended” mode. This prevents the module from trying to open UI dialogs or search for symbols interactively, which would cause an automated process to hang indefinitely.

6. Combine with Crash Report Client (CRC)

The CrashDebugHelper is most powerful when paired with a custom-hosted Crash Report Receiver. Once your receiver collects the .zip file from the client, use this module to unzip and analyze the contents programmatically on the server.

7. Verify String Formatting

When extracting call stacks, the module can return them in several formats. For maximum readability in task-tracking software (like Jira), use the “Human Readable” format settings to ensure that the module includes the full file path and line number for every entry in the call stack.

8. Monitor for “Missing Symbols”

If the module returns a call stack filled with UnknownFunction, it means it cannot find the matching .pdb. Implement a check in your tool to flag when a crash cannot be resolved, as this usually indicates a failure in your symbol storage pipeline.
