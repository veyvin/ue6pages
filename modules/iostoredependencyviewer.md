### AI Description & Usage Tips

e Viewer, which tracks high-level asset links, this module focuses on how data is physically bundled into .utoc and .ucas container files.

It is primarily used by technical artists and optimization engineers to identify “dependency bloat,” where a single small asset accidentally pulls in massive amounts of data into a specific IO chunk. This visibility is essential for the elimination of redundant data loading and the reduction of initial load times in modern Unreal Engine 5 projects.

Practical Usage Tips and Best Practices
1. Identify “Heavy” Dependency Chains

Use the viewer to find assets that act as “hubs.” If a single Blueprint or Material is pulling in hundreds of textures, it may be causing a massive IO bottleneck. Identifying these hubs facilitates the elimination of “monolithic” assets in favor of more modular, lazily-loaded components.

2. Analyze Chunk Fragmentation

The IO Store packages data into chunks. Use the module to verify that assets frequently used together are stored in the same or adjacent chunks. Proper data locality leads to the elimination of excessive disk seeking, which is critical for maintaining high performance on older HDD-based consoles or slower mobile storage.

3. Audit Package “Elimination” and Stripping

When you exclude a folder from a build, use the IOStoreDependencyViewer to ensure those assets—and their unique dependencies—are actually gone. If a “removed” asset’s dependencies still appear in the container, it indicates a “hard reference” from another system that must be found to ensure the total elimination of that data from the shipping build.

4. Trace “Invisible” Dependencies

Some dependencies are not obvious in the Content Browser, such as those generated by the cooker or specific shader permutations. The viewer exposes these low-level IO links, aiding in the elimination of “mystery” memory usage where the game appears to be loading more data than the level seems to require.

5. Optimize for Zen Loader

Since the Zen Loader uses a pre-computed dependency graph, any circular or overly complex dependency chain can slow down the initial boot sequence. Using the viewer to simplify these graphs facilitates the elimination of “blocking” operations during the engine’s startup phase.

6. Debug Incremental Cook Issues

If an incremental cook is failing or taking too long, the IOStoreDependencyViewer can show if a minor change in one asset is causing a massive “ripple effect” across the IO Store containers. Identifying these volatile assets leads to the elimination of unnecessarily long iteration times for the build team.

7. Combine with “Size Map” for Context

While the Size Map shows how large an asset is, the IOStoreDependencyViewer shows why that asset is being pulled into a specific container. Using both tools together is a best practice for the elimination of build size bloat, as you can see both the “what” and the “how” of your project’s storage footprint.

8. Verify Build.cs for Custom Tooling

If you are building a custom editor tool to automate dependency audits, ensure you include "IOStoreDependencyViewer" in your Editor.Build.cs. This allows you to programmatically access the dependency graph data, assisting in the elimination of manual, repetitive auditing tasks for large-scale production environments.
