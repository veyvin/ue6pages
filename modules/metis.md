### AI Description & Usage Tips

brary, a highly optimized system for partitioning unstructured graphs and meshes. In the context of UE5, its primary purpose is to drive the back-end of Nanite and ProxyLOD systems.

When you import a high-poly mesh, the engine must break it down into manageable “clusters” (typically 128 triangles each) that can be swapped in and out based on distance. The Metis module provides the mathematical algorithms to find the most efficient “seams” to cut the mesh, ensuring that clusters are logically grouped and that the hierarchical levels of detail (HLODs) are geometrically sound.

Practical Usage Tips and Best Practices
1. Optimize Nanite Cluster Borders

The Metis algorithm works best when the input geometry is “clean.”

Best Practice: Ensure your high-poly source meshes are manifold (watertight) where possible. While Metis can handle open edges, manifold geometry allows for more logical partitioning, which helps eliminate visible “cracks” or artifacts between Nanite clusters during LOD transitions.
2. Monitor “Nanite Stats” for Partitioning Efficiency

If a mesh is partitioned poorly, you will see a high number of tiny, inefficient clusters.

Action: Use the console command NaniteStats to inspect cluster density. If you see an abnormal number of clusters for the triangle count, try slightly simplifying the base mesh in your modeling software. This helps Metis generate better partitions and helps you eliminate excessive memory overhead on the GPU.
3. Use for Custom ProxyLOD Generation

If you are using the ProxyLOD plugin to merge massive city blocks into a single mesh, Metis is the engine under the hood.

Tip: Adjust the “Merge Distance” and “Hard Angle Threshold” in the ProxyLOD settings. These parameters change the graph that Metis sees; fine-tuning them helps eliminate “geometry soup” where distinct buildings accidentally fuse into unreadable blobs.
4. Balance Triangle Counts for HLODs

Metis partitions graphs based on connectivity. If a level has many disconnected floating meshes, partitioning becomes difficult.

Action: Try to group related small props into a single Actor or Blueprint before generating HLODs. This gives the Metis partitioner a more cohesive graph to work with, eliminating fragmented draw calls and improving the efficiency of the resulting HLOD clusters.
5. Understand the C++ Dependency

Metis is a low-level developer tool and is not intended for runtime gameplay logic.

Best Practice: Only include the "Metis" module in your Editor or Developer build modules within your *.Build.cs. This module is not available in “Shipping” builds, and excluding it correctly will eliminate packaging errors for your final game.
6. Improve Virtual Shadow Map (VSM) Performance

VSMs rely heavily on the cluster data generated by Metis.

Tip: If your shadows are flickering, it may be due to clusters being too small or overly complex at the edges. Ensuring your meshes have consistent triangle density allows Metis to create uniform clusters, which helps eliminate shadow depth bias issues and flickering in VSM.
7. Leverage for Data-Driven Graph Splitting

If you are building a custom engine tool that needs to split a large network (like a massive road system or a complex AI navigation graph) into chunks:

Action: Access the Metis API directly via C++ to partition your custom graphs. This utilizes the same battle-tested logic used for Nanite to eliminate the need for you to write your own complex graph-splitting algorithms.
8. Verify DDC (Derived Data Cache) Health

Since the Metis partitioning process happens during the “Building Nanite” phase, the results are stored in the DDC.

Best Practice: If you notice that Nanite meshes look different on different team members’ machines, clear your DDC. This forces the Metis module to re-partition the geometry, eliminating potential inconsistencies caused by corrupted or outdated cached data.
